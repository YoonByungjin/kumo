
[Technical Deep Dive] 실시간 알림 시스템 (Notification System)
2. 핵심 기술 스택 및 아키텍처
- Backend: Java 21, Spring Boot 3.5.8, Spring Data JPA, Spring Security
- Frontend: JavaScript (ES6+), Fetch API, CSS3 (Theme-aware)
- Architecture: Event-driven Polling 기반의 RESTful 통신

3. 기술적 상세 구현 (Technical Implementation)

3.1. RESTful API 설계 및 데이터 정합성
- 동적 상태 관리: 알림의 읽음/안읽음 상태를 처리하기 위해 PATCH 메서드를 활용하여 리소스의 부분 업데이트를 수행, 불필요한 데이터 전송 최소화.
- 보안 중심 설계: Spring Security의 @AuthenticationPrincipal을 활용하여 사용자 인증 정보와 알림 소유권을 대조, 타인의 알림 접근을 원천 차단하는 데이터 접근 제어 로직 구현.
- 일괄 처리 기능: 알림 전체 읽음 및 전체 삭제 기능을 제공하여 사용자 편의성 증대 및 데이터베이스 대량 트랜잭션 처리 최적화.

3.2. 프론트엔드 UX 및 실시간성 확보
- Event-driven Polling: 서비스의 초기 단계임을 고려하여 WebSocket의 오버헤드 대신, 페이지 전환 및 특정 UI 액션 시점에 비동기(Fetch) 요청을 수행하는 효율적인 폴링 전략 채택.
- 다국어 대응 (i18n): 서버의 MessageSource와 클라이언트의 Locale 정보를 연동하여 한국어/일본어 환경에 맞는 알림 메시지를 동적으로 생성.
- Relative Time Formatting: 알림 생성 시간을 '방금 전', 'n시간 전' 등 상대적인 시간으로 변환하는 전용 로직을 구현하여 사용자 직관성 개선.

4. 기술적 의사결정: 왜 Polling인가?
   프로젝트 개발 기간(2개월)과 운영 효율성을 고려하여 WebSocket/SSE 대신 Polling 방식을 선택했습니다.

1. 개발 효율성: 표준 HTTP 프로토콜을 사용하여 별도의 프로토콜 설정 없이 빠른 MVP(Minimum Viable Product) 구현 가능.
2. 서버 자원 최적화: 수천 명의 사용자가 실시간 연결(Connection)을 유지해야 하는 WebSocket에 비해, 요청 시점에만 자원을 사용하는 Polling 방식이 초기 서버 인프라 비용 절감에 유리하다고 판단.
3. 데이터 정합성: 알림 데이터의 실시간성보다 정확한 '읽음 상태' 동기화가 더 중요하다고 판단하여 REST API 기반의 명확한 상태 관리에 집중.

5. 주요 성과 (Key Achievements)
- 사용자 경험 개선: 미열람 알림 개수를 실시간 뱃지(Badge)로 시각화하여 사용자의 재방문 유도.
- 확장성 고려: 향후 SSE(Server-Sent Events)로의 전환이 용이하도록 알림 생성 로직과 전송 로직을 서비스 레이어에서 분리하여 설계.
- UI/UX 완성도: 다크 모드 및 다양한 디바이스 환경에 대응하는 반응형 알림 드롭다운 UI 구현.
